Context for humans

The emergence of agents in code assistants has represented an enormous leap in AI-assisted software development.

The first thing that stands out is their ability to automatically iterate on their own responses, create and modify files, run commands in the terminal, etc. But for me, the skill that makes the difference is that they are capable of indexing and analyzing a project's code, including dependencies between files and relationships between components.

If until now tools like GitHub Copilot allowed us to work at the file level, with the emergence of agents (GitHub Copilot Agent Mode, Cursor, etc.) we have moved to working at the architecture level. Now AI not only assists us in writing/fixing parts of our code, but also helps us in building complete projects (and at a speed that sometimes causes vertigo).

But this increase in the complexity of developed tasks brings about something that was previously difficult to see: we are pushing the context window of the agents themselves to the limit.

If you're familiar with using GitHub Copilot Agent Mode or Cursor, you'll already know that to work in agent mode, you just need to open a new chat, select agent, and start the conversation. The issue is that the open chat has a context window, which if we exceed, context displacement begins to occur‚Äîbasically, the agent starts to forget the beginning of the conversation.

If the task is simple, there's no problem; the limit is rarely exceeded, and if it happens, simply opening a new chat is enough. But when building complete projects, how do we maintain conversation coherence and work quality?

For these cases, I use a strategy that I've designed from my experience with Cursor first, and with GitHub Copilot Agent Mode in recent weeks: extended context.

Extended context (using markdowns as memory)

This strategy is based on storing markdown files in a folder at the project root called context/

The central idea is to create a memory with project information that the LLM (or we) can refer to whenever it needs to know what point it's at and what it should do, and to divide the work into as many chats as necessary to avoid reaching the context window limit. This way, we maintain the coherence and quality of the work from beginning to end.

Work scheme:

Chat 1

  1. Prompt to create a synthetic expert in the project's technology: role, experience, technical stack, reasoning approach, and response format. Ending with a "don't write code yet".
  2. Prompt where we ask to:
    • analyze the project code
    • generate a report of its analysis
    • save it in .md format inside the context/ folder with a 00- prefix to the filename. If we're not entirely satisfied with the report, we can iterate on it until we are.
  3. Prompt where:
    • we indicate what we want to do (requirements)
    • ask to generate a master plan
    • save the master plan in .md format inside the context/ folder with a 01- prefix to the filename. As with the analysis, we can iterate on it until we are satisfied.

Chat 2

  1. Same prompt for creating synthetic expert
  2. Prompt asking to read and internalize the project analysis
  3. Prompt asking to read and internalize the master plan
  4. Prompt asking to execute point 1 of the master plan
  5. Normal iteration until point 1 is finished.
  6. Prompt asking to document the work developed in point 1. In a .md inside context/ following the current numbering

Chat n

  1. Same prompt for creating synthetic expert
  2. Prompt asking to read and internalize the project analysis
  3. Prompt asking to read and internalize the master plan
  4. Prompt asking to execute point n of the master plan
  5. Normal iteration until point n is finished
  6. Prompt asking to document the work developed in point n. Inside the documentation .md created previously

Advantages of extended context

This strategy offers several advantages:

  1. Avoids context loss: The LLM can consult these files when it needs to remember previous decisions.
  2. Continuity: You can close your computer and resume work days later without losing the thread.
  3. Collaboration: Other team members can easily understand the project status and continue it at any point.
  4. Automatic documentation: Upon completion, you already have structured documentation.
  5. Facilitates task division: Perfect for agile methodologies and sprints.

Final reflection

Just as humans need to write things down so we don't forget them, LLMs also benefit from having their own notes (if they're in markdown, even better‚Äîthey love markdown!). It's ironic that in this imitation of human thought, we have not only replicated its successes but also its limitations. Although with the race for expanding the context window that exists, perhaps in the future these types of strategies won't be necessary.

This way of working has worked especially well for me in developing my MCP Claude Talk to Figma and especially in the refactoring I did during Easter week (I had already refined the flow). At first, it may feel like we're making the work more complex, but in the end, the time and frustration savings we achieve are completely worth it.
